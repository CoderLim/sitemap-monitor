# 关键词快捷时间范围设计

日期：2026-07-23  
状态：已确认

## 背景

Dashboard「新增关键词」与「Google Trends」目前只能按单日查看 `reports/<date>-<site>.json` 中的新增词。需要支持三个快捷范围：最近一天、最近一周、最近一个月。

## 目标

- 用三个快捷范围替代日期下拉，快速查看近期新增词
- 「新增关键词」与「Google Trends」共用同一套范围 + 站点筛选
- 多日范围内同一站点同一关键词去重，只保留首次出现

## 非目标

- 自定义起止日期
- 后端 / Worker 聚合 API
- 修改抓取逻辑或报告文件格式
- 按北京时间改写报告日期戳（仍以现有 UTC 报告日为准）

## 交互

- 两个页面共用筛选状态：`range`（`1d` | `7d` | `30d`）+ `siteId`
- 去掉原「日期」下拉，改为三个快捷项：**最近一天 / 最近一周 / 最近一个月**
- 默认：`1d`（最近一天）
- 站点筛选不变：全部 / 单站
- 面板标题可从「按日新增词」调整为体现范围（如「近期新增词」）

## 范围计算

- 终点：`/api/reports/dates` 返回列表中最新的有报告日期（UTC 报告日）
- 窗口：从终点往前包含 N 个日历日（N = 1 / 7 / 30），即日期字符串落在 `[end - (N-1) days, end]`
- 实际请求：只拉取该窗口内**确实存在**的报告日期（缺跑的天跳过）
- 「最近一天」= 最新有报告的那一天（不是「日历上的今天」）

## 数据流（纯前端）

1. 启动 / 刷新时加载站点、日期列表、运行状态、异常（现有逻辑）
2. 根据 `range` 从日期列表筛出窗口内日期
3. 对每个日期：
   - 站点 = 全部：先 `listReports(date)`，再并行 `getReport(date, site)`
   - 站点 = 单站：直接 `getReport(date, site)`（无则跳过）
4. 合并所有报告明细后做去重，驱动关键词表与 Trends 分组

不新增 API；沿用现有 `/api/reports/dates`、`/api/reports`。

## 去重与展示

- 键：`site_id + keyword`
- 规则：只保留**首次出现**（报告日期最早）；URLs 取该首次记录
- 表格列：关键词 |（全部站点时）站点 | **首次日期** | 对应 URL
- Trends：对去重后的关键词按现有规则每 5 个一组生成链接
- 「复制全部」：复制去重后的唯一关键词列表

## 边界与错误

| 情况 | 行为 |
|------|------|
| 窗口内无报告日期 | 空状态：「该范围内暂无报告」 |
| 有报告但无新词 | 沿用：「无新增关键词」 |
| 某日/某站请求失败 | 跳过失败项，其余照常；可顶部提示「部分日期加载失败」 |
| 切换范围或站点 | 重新拉取并显示 loading |

## 实现范围

- 主要改动：`dashboard/src/App.tsx`、少量 `dashboard/src/styles.css`
- 可选：抽出日期窗口计算与去重小函数（便于单测）
- README：功能说明改为一行（按快捷范围查看新增词）
- **不改**：Python API、Worker、CLI、报告 schema

## 测试

- 若抽出纯函数：为窗口计算与去重补单元测试
- 手工验证：
  - 三个范围在关键词页与 Trends 页结果一致（共用状态）
  - 切换站点后范围不变
  - 多日重复词只出现一次，且「首次日期」正确
  - 无报告 / 无新词 / 部分失败时的空态与提示
